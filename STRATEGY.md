# STRATEGY

## Core strategy
Because implementation speed is no longer the bottleneck, Hashapp should be developed with a **build broad, demo narrow** approach.

## Why
We can use agents to generate broad product surface area quickly. That means our scarce resource is no longer raw implementation throughput. It is:
- judgment
- clarity
- taste
- positioning
- emotional punch

## Build broad
Use agents and parallel execution to build more of the product surface than a normal team could in the same time.

Candidate surfaces:
- Home dashboard
- Activity feed
- Receipt detail
- Payees / trusted destinations
- Agent identity and profile pages
- Policy presets / personas
- Approvals
- Error and fail states
- Fleet / multi-agent overview
- Session setup or permission flows

## Demo narrow
Present a single, crystal-clear story in the live demo.

The demo should center on:
- one human
- one named agent
- one concrete use case
- one rule set
- one approval moment
- one blocked moment
- one beautiful receipt feed

## Why this wins
- judges feel the product immediately
- the app still looks bigger than the demo
- we get credit for polish and scope without diluting the story
- the broad product surface makes the concept feel like a real company, not a one-off prototype

## Product center
Hashapp should not read as a generic wallet dashboard or an infra tool. It should read as a spending app for AI agents.

## Demo principle
Build the larger system, but never let the live demo lose its emotional center.
