# Onboarding

## Why it matters
Hashapp must not only look good after setup. It must teach a normal human, quickly and clearly:
- your money stays in your wallet
- Scout only gets scoped permissions
- you can approve, auto-approve, or block
- you can revoke at any time

If onboarding fails, the product feels scary.

## Core job
Get the user from zero to:
**“Scout can spend safely under my control.”**

## Recommended shape
Do not build a long technical wizard.
Do not start with dense Rules UI.
Do not overexplain protocols.

Hashapp onboarding should be a short trust-building flow.

## Four-step onboarding
### 1. Connect wallet
Purpose:
- establish that funds stay in the user’s wallet
- explain non-custodial model simply

Key copy:
- Your wallet stays yours
- Hashapp never takes custody

### 2. Meet Scout
Purpose:
- make Scout feel real, useful, and safe

Show:
- Scout avatar
- what Scout does
- what Scout can ask to buy
- what kind of value it creates

### 3. Set Scout’s first boundary
Purpose:
- give Scout a safe starting budget and policy

Show:
- first budget/allocation
- basic preset
- approval mode

This should feel like giving Scout an allowance, not configuring a wallet engine.

### 4. Review first request
Purpose:
- let the user experience one real permission moment

Show:
- Scout asks for one small purchase
- user approves, denies, or changes rule
- result lands in Activity

Then drop the user into the real app.

## What onboarding should teach
The user should leave onboarding understanding:
- Scout cannot drain the wallet
- I choose how much freedom Scout gets
- I can see what Scout did
- I can stop it instantly

## Product rules
- onboarding starts with trust, not settings
- one obvious next action per screen
- plain-English language only
- no protocol jargon in primary copy
- no fake complexity

## Nice future additions
- quick preset chooser
- emergency stop explanation
- “why this is safer than giving an agent your whole wallet” panel
